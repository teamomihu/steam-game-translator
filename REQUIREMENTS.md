# Steam 游戏汉化工具 - 需求规格书 v2.0

> 基于 GitHub 开源项目调研、社区精华帖经验整理
> 定位: 实时翻译覆盖 + 汉化补丁制作 双模式全覆盖平台
> 目标用户: 普通玩家 (一键翻译) + 汉化组/译者 (专业工作流)

---

## 一、竞品调研总结

### 1.1 核心竞品分析

| 项目 | Stars | 技术路线 | 优势 | 不足 |
|------|-------|---------|------|------|
| **LunaTranslator** | 11.2k | HOOK + OCR + 内嵌翻译 | 功能最全面，支持HOOK/OCR/模拟器，翻译引擎最丰富 | 主要面向视觉小说，非通用Steam游戏 |
| **XUnity.AutoTranslator** | 3.1k | Unity HOOK + 资源替换 | Unity游戏通杀，支持贴图翻译，翻译缓存机制成熟 | 仅限Unity引擎游戏 |
| **UGTLive** | 74 | 屏幕OCR + LLM翻译 | AI驱动，引擎无关，支持26种语言，完全本地运行 | 需NVidia RTX GPU，翻译延迟较高 |
| **XUnity对接软件** | 15 | XUnity桥接 + AI LLM | 专为中文优化，支持术语表，并发 | 依赖XUnity，仅Unity游戏 |

### 1.2 竞品缺口 (我们的机会)

| 缺口 | 详细说明 |
|------|---------|
| **无通用汉化补丁制作能力** | LunaTranslator和UGTLive都是实时翻译，不能产出可分发的汉化补丁 |
| **引擎覆盖不全** | XUnity仅Unity，无人覆盖RPG Maker/UE/Godot全家桶 |
| **无 Steam 平台集成** | 没有利用 Steam Workshop 分发、SteamDB 引擎检测等能力 |
| **图片翻译薄弱** | XUnity有贴图替换但需手动，其他竞品几乎没有图片翻译 |
| **无增量更新能力** | 游戏更新后翻译全部失效，没有diff/增量同步机制 |

### 2.2 模式A: 实时翻译覆盖 (普通玩家)

#### M1: 文本提取引擎

```
优先级: P0 (必须)
```

**1. 屏幕OCR模式 (通用，引擎无关 -- 核心能力)**
- 支持框选屏幕区域进行实时OCR
- 支持的OCR引擎 (按优先级):
  - **本地引擎**: PaddleOCR (推荐，中日英精度高), EasyOCR, Windows OCR, docTR, MangaOCR (日文竖排专用)
  - **云端引擎**: Google Cloud Vision, 百度OCR
- 自动检测文本变化，变化后触发翻译 (防抖机制: 等画面稳定1秒后再OCR)
- 支持全屏/窗口/区域三种截取模式
- 支持竖排文本识别 (日文常见)
- 智能区域记忆: 记住上次使用的区域位置

**2. HOOK文本提取 (引擎特定，高精度)**
- 针对Unity游戏: Hook UGUI / TextMeshPro / NGUI / IMGUI / FairyGUI
- 针对Ren'Py游戏: 提取 .rpy 脚本中的对话文本
- 针对RPG Maker MV/MZ: Hook nw.js 中的 Window_Message 等文本渲染类
- 通用 Win32 TextOut / DrawText / ExtTextOut API Hook
- 选择正确HOOK点的辅助界面 (参考LunaTranslator的特殊码选择)

**3. 智能模式选择**
- 启动时自动检测游戏引擎 (通过文件特征/Steam信息)
- 自动推荐最佳提取模式 (HOOK优先，OCR兜底)
- 用户可手动切换

#### M2: 翻译引擎

```
优先级: P0 (必须)
```

**1. 多翻译源支持**

| 类型 | 引擎 | 特点 | 推荐场景 |
|------|------|------|---------|
| **LLM翻译** | OpenAI, Gemini, DeepSeek, Claude | 质量最高，支持上下文和术语表 | 对话/剧情翻译 |
| **LLM本地** | Ollama, llama.cpp, Sakura模型 | 免费，隐私安全 | 离线/隐私敏感 |
| **专业翻译** | DeepL, Google, 百度, 有道 | 速度快，稳定 | UI/菜单快速翻译 |
| **离线翻译** | Sugoi Translator, Argos Translate | 完全离线 | 无网络环境 |

**2. 翻译质量优化**
- 术语表系统: 角色名/地名/技能名专有名词一致性 (支持导入CSV)
- 上下文窗口: 发送最近N条对话作为上下文 (可配置，默认5-10条)
- LLM Prompt 模板: 可自定义翻译风格 (直译/意译/口语化)
- 翻译缓存: 内存LRU + SQLite磁盘持久化，相同原文不重复请求
- 预处理管道: 变量占位符保护 ({name}/%d)、HTML标记保护、正则过滤
- 后处理管道: 宽字符标准化、HTML还原、长度适配(自动换行/缩放)
- 回退机制: 主翻译源失败自动切换备用源

#### M3: 显示与覆盖层

```
优先级: P0 (必须)
```

- 透明覆盖窗口，悬浮在游戏画面上方
- 支持点击穿透 (passthrough模式)，不影响游戏操作
- 文字颜色/背景自动适配 (检测背景色，高对比度显示)
- 多种显示模式:
  - 覆盖原文 (背景修复 + 渲染译文)
  - 译文浮窗 (独立窗口显示译文)
  - 原文+译文对照 (双行显示)
- 一键切换原文/译文 (快捷键)
- 窗口位置/大小记忆
- 全屏游戏兼容: 引导使用窗口化/无边框模式

#### M4: 用户交互

```
优先级: P0 (必须)
```

- 全局快捷键系统 (可自定义):
  - `Ctrl+Shift+T`: 截图翻译 (框选)
  - `Ctrl+Shift+S`: 开始/停止实时翻译
  - `Ctrl+Shift+D`: 切换原文/译文
  - `Ctrl+Shift+R`: 重新翻译当前内容
  - 支持手柄按键绑定
- 系统托盘图标 (右键菜单快速操作)
- 拖拽调整翻译区域
- 首次使用引导向导 (3步完成配置)

### 2.3 模式B: 汉化补丁制作 (汉化组/译者)

#### M5: 游戏引擎适配器 (资源解包/回注)

```
优先级: P0 (必须)
```

这是补丁制作的核心。针对Steam上主流引擎逐一适配:

**1. Unity 引擎适配器**
- 解包: AssetBundle / resources.assets / level 文件
- 工具依赖: AssetStudio, UABE (Unity Asset Bundle Extractor)
- 文本提取: 从 MonoBehaviour / TextAsset / Font 中提取字符串
- 字体注入: 替换/追加中文字体资源 (思源黑体)
- 贴图替换: dump原始贴图 → 翻译 → 通过hash匹配替换
- 回注: 修改后的资源写回 AssetBundle
- 框架: 通过 BepInEx / MelonLoader 注入运行时翻译插件
- IL2CPP: 使用 BepInEx 6 for IL2CPP / cpp2il 处理
- **关键坑**: 不同Unity版本的AssetBundle格式不同，需要版本检测

**2. Ren'Py 引擎适配器**
- 解包: .rpa 归档文件解包 (unrpa)
- 文本提取: 解析 .rpy / .rpyc 脚本，提取对话和选项文本
- 翻译文件: 生成 Ren'Py 标准翻译目录 (game/tl/chinese/)
- 字体: 修改 gui.rpy 中的字体配置
- 回注: 直接放入 game/tl/ 目录，Ren'Py原生支持多语言
- **关键坑**: .rpyc 是编译后的字节码，需要反编译 (unrpyc)

**3. RPG Maker 适配器**
- RPG Maker MV/MZ (主流):
  - 文本位置: data/*.json (Map, CommonEvents, System, Actors等)
  - 脚本位置: js/plugins/*.js
  - 字体: 替换 fonts/ 目录下的字体文件
  - 插件文本: 部分插件有独立的文本配置
- RPG Maker VX Ace:
  - 文本位置: Data/*.rvdata2 (Marshal序列化)
  - 脚本位置: Data/Scripts.rvdata2
  - 需要Ruby环境解析
- RPG Maker XP:
  - 文本位置: Data/*.rxdata
- **关键坑**: MV/MZ的JSON中文本散布在各种嵌套结构中，需要深度遍历; 插件自定义的文本格式无统一标准

**4. Unreal Engine 适配器**
- 文本位置: .locres 本地化资源文件
- 解包工具: UnrealPak / UModel / FModel
- 文本格式: UE的本地化系统使用 FText, 存储在 .locres / .uasset 中
- 国际化: UE4/5原生支持 Localization Dashboard，可生成 .po 文件
- **关键坑**: UE打包后资源高度压缩，且不同项目的pak密钥不同;
  部分游戏使用自定义本地化方案而非UE原生系统

**5. Godot 引擎适配器**
- 文本位置: .tscn (场景文件), .tres (资源文件), .gd (脚本)
- 国际化: Godot原生支持 .csv / .po 格式翻译文件
- 解包: .pck 文件解包 (godotpcktool)
- **关键坑**: Godot 3.x和4.x的文件格式有较大差异

**6. GameMaker 适配器**
- 解包: data.win 文件解包 (UndertaleModTool)
- 文本: 从字符串表中提取
- **关键坑**: GameMaker Studio 2 使用 .yyz 格式，需要不同解包方式

**7. 通用文本格式适配**
- 自动检测并处理: JSON, CSV, XML, YAML, PO/MO, INI, TXT
- SRT/ASS/VTT 字幕文件翻译
- Lua脚本中的字符串提取 (很多自研引擎使用Lua)
- 二进制文件中的字符串扫描 (strings风格，作为最后手段)

#### M6: AI批量翻译 + 人工校对工作台

```
优先级: P0 (必须)
```

**1. 批量翻译**
- 导入提取的文本文件 (CSV/JSON/PO)
- 一键AI批量翻译 (支持并发，可配置并发数)
- 翻译进度条 + 预计剩余时间
- 断点续翻 (中断后从上次位置继续)
- 翻译结果自动保存

**2. 人工校对工作台**
- 三栏界面: 原文 | 机翻 | 人工校对
- 快捷键:
  - `Enter/Tab`: 采纳机翻，下一条
  - `Ctrl+Enter`: 保存修改，下一条
  - `Ctrl+R`: 重新AI翻译当前条
  - `Ctrl+F`: 查找/替换
  - `Ctrl+G`: 添加到术语表
- 状态标记: 未翻译 / 机翻待审 / 已审核 / 有问题
- 筛选器: 按状态、长度、包含关键词筛选
- 上下文预览: 显示当前文本在游戏中的上下文 (前后5条)
- "边玩边校对"模式: 游戏运行中，看到有问题的翻译，快捷键弹出校对窗口

**3. 术语表管理**
- 全局术语表 + 项目术语表
- 术语条目: 原文 | 译文 | 备注 | 词性(人名/地名/技能/物品)
- 一键从文本中提取疑似专有名词 (NER命名实体识别)
- 术语表导入/导出 (CSV)
- 翻译时自动注入术语表到LLM prompt

#### M7: 汉化补丁打包与分发

```
优先级: P1 (重要)
```

**1. 补丁打包**
- 自动对比原始文件和翻译文件，生成差异补丁
- 补丁格式: 独立文件夹 (直接覆盖) 或 安装器 (.exe)
- 包含: 翻译文件 + 中文字体 + 安装说明
- 补丁元数据: 工具版本、游戏版本、翻译进度、翻译者信息

**2. Steam Workshop 集成**
- 将汉化补丁打包为 Workshop mod (如果游戏支持)
- 自动填充 Workshop 元数据 (标题/描述/标签)
- 一键上传到 Steam Workshop (通过 Steamworks SDK)

**3. 其他分发渠道**
- 导出为 Nexus Mods 兼容格式
- 导出为 GitHub Release 格式
- 生成安装说明文档

#### M8: 游戏更新增量同步

```
优先级: P1 (重要)
```

- 检测游戏更新 (对比文件hash/版本号)
- 自动识别新增/修改/删除的文本
- 差异报告: 显示需要翻译的新增文本和需要检查的修改文本
- 保留已有翻译，仅标记需要更新的部分
- 通知翻译者更新工作量估算

### 2.4 通用模块 (两种模式共用)

#### M9: Steam 平台集成

```
优先级: P1 (重要)
```

**1. 游戏识别**
- 自动扫描Steam库目录，列出已安装的游戏
- 通过 Steam appmanifest 文件读取游戏信息 (appid, name, installdir)
- 通过 appid 查询 SteamDB/Steam Store 获取:
  - 游戏引擎类型 (Unity/UE/RPG Maker 等)
  - 已支持的语言列表
  - 是否有官方中文
  - 是否支持 Workshop

**2. 游戏启动集成**
- 从工具内直接启动游戏 (调用 steam://rungameid/)
- 启动时自动附加翻译模式 (HOOK注入或OCR启动)
- 游戏退出检测

**3. 社区翻译共享 (远期)**
- 翻译结果上传/下载 (云端翻译记忆库)
- 玩家投票评价翻译质量
- 热门游戏的社区翻译排行

#### M10: 图片/贴图翻译

```
优先级: P1 (重要)
```

**1. 实时模式: OCR + 覆盖渲染**
- 识别画面中的图片文字 (UI按钮、菜单、物品图标上的文字)
- 在覆盖层上渲染对应位置的中文翻译
- 适用于: 临时玩游戏，不需要修改游戏文件

**2. 补丁模式: 贴图替换 (实际可行的方案)**
- **手动替换流程** (主力方案):
  - dump游戏贴图 → 列出含文字的贴图 → 图片编辑工具中翻译 → 通过hash替换
  - 提供批量dump/替换工具
  - 参考XUnity的TextureDirectory/TextureHashGenerationStrategy
- **AI辅助替换** (增强方案):
  - OCR识别贴图中的文字区域和内容
  - 使用AI图像生成/编辑 (Stable Diffusion inpainting) 擦除原文
  - 渲染中文译文到擦除区域
  - 人工确认后写回
- **关键限制**: 
  - 压缩格式贴图 (DXT/BC7/ASTC) 需要解压→处理→重压缩
  - 艺术字/特效字(描边/渐变/发光) AI无法完美复制，需手动PS
  - 此功能定位为"辅助工具"而非"全自动"

**3. 字体管理**
- 内置中文字体: 思源黑体 (UI), 思源宋体 (正文)
- 字体注入工具: 自动将中文字体注入游戏资源
- 字体大小自适应: 根据文本框大小自动缩放
- 备用字体链: 缺字自动fallback到下一个字体

#### M11: 翻译项目管理

```
优先级: P1 (重要)
```

- 按游戏创建翻译项目
- 项目数据: 游戏信息、引擎类型、翻译进度、术语表、翻译文件
- 翻译进度追踪:
  - 总文本数 / 已翻译 / 已审核 / 有问题
  - 按文件/章节的进度统计
  - 进度可视化 (进度条/饼图)
- 翻译记忆库 (TM): 跨项目复用相同翻译 (相同引擎的通用UI文本)
- 导入/导出: CSV, JSON, PO, XLIFF, TMX
- 版本历史: 记录每条翻译的修改历史

### 3.2 引擎适配器Plugin架构

```python
# 每个引擎适配器实现统一接口
class EngineAdapter(ABC):
    """游戏引擎适配器基类"""
    
    @abstractmethod
    def detect(self, game_path: str) -> float:
        """检测游戏是否使用该引擎，返回置信度 0-1"""
        pass
    
    @abstractmethod
    def extract_texts(self, game_path: str) -> List[TextEntry]:
        """提取所有可翻译文本"""
        pass
    
    @abstractmethod
    def inject_translations(self, game_path: str, translations: Dict) -> None:
        """将翻译回注到游戏文件"""
        pass
    
    @abstractmethod
    def extract_textures(self, game_path: str) -> List[TextureEntry]:
        """提取含文字的贴图"""
        pass
    
    @abstractmethod
    def inject_font(self, game_path: str, font_path: str) -> None:
        """注入中文字体"""
        pass
    
    @abstractmethod
    def create_patch(self, game_path: str, output_path: str) -> None:
        """打包汉化补丁"""
        pass
    
    def get_file_fingerprint(self, game_path: str) -> str:
        """获取游戏文件指纹，用于检测更新"""
        pass
```

### 3.3 技术选型

| 组件 | 推荐方案 | 理由 |
|------|---------|------|
| **主语言** | Python 3.11+ (核心) + C/C++ (HOOK/性能) | Python生态最丰富，C++做底层注入 |
| **GUI框架** | PySide6 (Qt6) | 开源许可(LGPL)，覆盖层支持好，跨平台 |
| **OCR** | PaddleOCR (主) + MangaOCR (日文竖排) + Windows OCR (备用) | PaddleOCR精度和速度最优 |
| **截图** | mss (高速截图) + Win32 API (窗口截取) | mss极快(>30fps)，Win32做精确窗口定位 |
| **覆盖层** | Win32 Layered Window (WS_EX_LAYERED + WS_EX_TRANSPARENT) | 最佳穿透+渲染性能 |
| **翻译API** | httpx (异步) + aiohttp | 异步并发，性能好 |
| **缓存** | SQLite (持久化) + cachetools LRU (内存) | 轻量，零依赖 |
| **HOOK** | frida (通用注入) + ctypes/pymem (轻量读写) | frida跨平台通用 |
| **资源解包** | UnityPy (Unity), unrpa (Ren'Py), 自研 (RPGMaker JSON) | 成熟开源库 |
| **图片处理** | OpenCV + PIL/Pillow + (可选) Stable Diffusion | CV做基础处理，SD做高级修复 |
| **打包分发** | Nuitka (编译) 或 PyInstaller | Nuitka性能更好 |
| **编码检测** | chardet / cchardet | 自动处理 Shift-JIS 等编码 |

## 四、各游戏类型应对策略

不同类型的Steam游戏有不同的翻译挑战:

### 4.1 日式RPG / 视觉小说

| 特征 | 策略 |
|------|------|
| 文字量巨大 (10万-100万字) | 批量翻译 + 增量校对; LLM上下文窗口保持剧情连贯 |
| 对话树/分支选项 | 按对话树结构组织翻译，保持分支上下文 |
| 角色语气差异 | LLM prompt中说明角色语气 (参考术语表中的角色备注) |
| 竖排日文 | MangaOCR + 竖排渲染 |
| 文字演出(逐字显示) | 防抖等待文字显示完成再翻译 |
| 存档兼容 | 翻译时不修改游戏逻辑代码，仅替换文本资源 |

### 4.2 欧美3A / 独立游戏

| 特征 | 策略 |
|------|------|
| 引擎多样 (UE/Unity/自研) | OCR通用模式为主，引擎检测后切换HOOK |
| 过场动画字幕 | SRT/字幕文件提取翻译 |
| 大量UI图片文字 | 贴图替换 (优先手动PS，AI辅助) |
| 3D空间浮动文字 | HOOK渲染函数或OCR (效果有限) |
| 多人在线部分 | 仅翻译单机UI部分，在线聊天不翻译 (避免反作弊) |

### 4.3 RPG Maker 游戏

| 特征 | 策略 |
|------|------|
| 数量极大 (Steam上数千款) | RPG Maker适配器优先级高 |
| 格式统一 | MV/MZ全是JSON，可批量处理 |
| 插件文本 | 自动扫描js/plugins/中的文本 |
| 图片事件 | 事件中显示图片的文字需要替换图片 |
| 加密归档 | 部分使用rpg_core加密，需解密 |

### 4.4 策略/模拟/沙盒游戏

| 特征 | 策略 |
|------|------|
| 大量动态生成文本 | 术语表覆盖核心词汇，翻译缓存应对重复 |
| 数值+文本混合 | 正则保护数值部分 |
| 界面复杂/文字密集 | OCR区域智能分割; 分优先级翻译(关键UI先翻译) |
| Mod支持好 | 利用游戏Mod系统分发翻译 |

### 4.5 快节奏动作游戏

| 特征 | 策略 |
|------|------|
| 文字出现短暂 | OCR实时模式效果差 → 优先用HOOK或补丁模式 |
| 文字量少 | 手动翻译+贴图替换更实际 |
| 性能敏感 | OCR降频到1fps; 缓存命中率是关键 |

## 五、从社区总结的常见坑与解决方案

### 5.1 OCR相关

| 问题 | 详细描述 | 解决方案 |
|------|---------|---------|
| **复杂背景干扰** | 游戏纹理/特效导致OCR误识别 | 图像预处理: 二值化、去噪、对比度增强; 深度学习OCR模型 |
| **竖排文本** | 日文竖排排版 | MangaOCR专门处理; PaddleOCR竖排模式 |
| **特殊字体/艺术字** | 游戏标题/技能名艺术字体 | 提供手动输入fallback; 艺术字走贴图替换 |
| **文字闪烁/滚动** | 文字逐字显示或滚动 | 防抖: 等文字稳定1秒后再OCR |
| **小字/模糊字** | 低分辨率/缩放导致模糊 | 截图放大2-4x再OCR; 超分辨率预处理 |
| **全屏独占模式** | 无法截取独占全屏画面 | 引导用户使用无边框窗口化; 或HOOK D3D/Vulkan截图 |

### 5.2 翻译质量相关

| 问题 | 详细描述 | 解决方案 |
|------|---------|---------|
| **专有名词不一致** | 角色名/地名每次翻译不同 | 术语表 + LLM prompt注入术语表 |
| **丢失变量/格式** | `{name}`被翻译成中文 | 预处理保护占位符，翻译后还原 |
| **断句错误** | 长文本截断翻译，语义断裂 | 按标点/段落智能分段 |
| **上下文缺失** | 单句翻译歧义多 | 上下文窗口 (最近5-10条) |
| **过度翻译** | 数字/代码被翻译 | 正则过滤 + 源语言检测 |
| **API限流** | 请求过多被封IP | 缓存 + batch + 限频 + 备用源 |
| **LLM幻觉** | LLM编造不存在的翻译 | 后处理检测: 译文长度异常/原文丢失 → 重试或降级到传统翻译 |

### 5.3 显示渲染相关

| 问题 | 详细描述 | 解决方案 |
|------|---------|---------|
| **中文字体缺失** | 游戏无中文字体→乱码 | 内嵌字体; OverrideFont机制 |
| **文本框溢出** | 译文比原文长 | 自动缩放字体; 自动换行 |
| **DPI缩放偏移** | Windows缩放导致位置错 | DPI-aware; 物理像素坐标 |
| **多显示器** | 游戏在副屏，覆盖层在主屏 | 检测游戏窗口所在显示器 |

### 5.4 工程/兼容性相关

| 问题 | 详细描述 | 解决方案 |
|------|---------|---------|
| **编码混乱** | Shift-JIS/EUC-JP/GBK等 | chardet自动检测; 统一转UTF-8 |
| **反作弊** | EAC/VAC拦截HOOK | OCR模式不注入，完全兼容; HOOK前警告用户 |
| **IL2CPP** | Unity IL2CPP游戏HOOK难 | BepInEx 6 for IL2CPP |
| **游戏更新** | 更新后翻译失效 | hash指纹检测 + 增量diff |
| **存档兼容** | 翻译修改导致存档不可用 | 仅替换文本/贴图资源，不修改逻辑 |
| **DRM** | Denuvo等DRM干扰 | OCR模式不受影响; HOOK可能需要等破解 |
| **Python环境** | 依赖冲突 | 内嵌Python环境; Nuitka编译; 预打包 |
